Introduction

OpenClaw is an autonomous operator agent designed to run 24/7 on a VPS — handling tasks, monitoring systems, and executing workflows without human intervention. This guide walks you through the complete setup: from provisioning a Hetzner VPS to systemd service configuration, nginx reverse proxy, SSL, and firewall hardening. By the end, you'll have OpenClaw running as a resilient system service that survives reboots, recovers from crashes, and stays online indefinitely.

What you'll need: A VPS with Ubuntu 22.04+ (recommended), a domain name with an A record pointing to your server IP, and 10–15 minutes of focused setup time.

Prerequisites

Before you start, verify these requirements:

  • VPS: Ubuntu 22.04 LTS (or 20.04 LTS) with root or sudo access. Hetzner CX21 or better recommended. See our VPS comparison guide for recommendations.
  • Domain: A domain or subdomain (e.g., openclaw.yourdomain.com) with an A record pointing to your VPS IP. Propagation can take up to 24 hours — set this up before starting.
  • SSH access: Connect to your VPS via SSH. Use key-based authentication; password auth should be disabled after setup.
  • Firewall: Ports 22 (SSH), 80 (HTTP), 443 (HTTPS) must be accessible.

Step 1: Configure Firewall

Start by hardening your VPS with UFW (Uncomplicated Firewall). This blocks all non-essential ports while allowing SSH, HTTP, and HTTPS.

# Check UFW status
sudo ufw status

# Allow SSH (port 22) — don't lock yourself out
sudo ufw allow 22/tcp comment 'SSH'

# Allow HTTP and HTTPS
sudo ufw allow 80/tcp comment 'HTTP'
sudo ufw allow 443/tcp comment 'HTTPS'

# Enable UFW
sudo ufw enable

# Verify rules
sudo ufw status verbose
Critical: Ensure you have active SSH access before enabling UFW. If you enabled it without allowing port 22, you'd lock yourself out of the server. Use sudo ufw allow 22/tcp first.

Step 2: Install Dependencies

Update your system and install Node.js 22 LTS, nginx, and Certbot.

# Update package lists and upgrade system
sudo apt update && sudo apt upgrade -y

# Install Node.js 22 LTS
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs

# Verify Node.js and npm versions
node --version    # Should output v22.x.x
npm --version     # Should output 10.x.x

# Install nginx
sudo apt install -y nginx

# Verify nginx
nginx -v          # Should output nginx 1.22.x or higher

# Install Certbot for Let's Encrypt SSL
sudo apt install -y certbot python3-certbot-nginx

Step 3: Create a Dedicated User

Run OpenClaw under a non-root user. This limits damage if the application is compromised.

# Create a user named 'openclaw'
sudo adduser openclaw

# Grant sudo privileges (optional — only if you need it)
sudo usermod -aG sudo openclaw

# Switch to the openclaw user to verify
sudo su - openclaw
exit

Step 4: Install OpenClaw

Log in as the openclaw user and install the OpenClaw package.

# Switch to openclaw user
sudo su - openclaw

# Install OpenClaw globally via npm
npm install -g openclaw

# Verify installation
openclaw --version

If you prefer installing from source or a specific Git repository, clone it and install dependencies:

# As openclaw user
cd ~
git clone https://github.com/your-repo/openclaw.git
cd openclaw
npm install
npm run build   # if build step required

Step 5: Create the Systemd Service

Create a systemd unit file so OpenClaw starts on boot, restarts on crash, and runs in the background.

# Create the service file
sudo nano /etc/systemd/system/openclaw.service

Paste the following configuration:

[Unit]
Description=OpenClaw Autonomous Agent
Documentation=https://openclaw.dev/docs
After=network.target network-online.target
Wants=network-online.target

[Service]
Type=simple
User=openclaw
Group=openclaw
WorkingDirectory=/home/openclaw
# Adjust the ExecStart path if you installed differently
ExecStart=/usr/bin/node /usr/lib/node_modules/openclaw/index.js
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal
SyslogIdentifier=openclaw
Environment=NODE_ENV=production
Environment=PORT=3000

# Security hardening
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ReadOnlyPaths=/
ReadWritePaths=/home/openclaw
CapabilityBoundingSet=

[Install]
WantedBy=multi-user.target

Save and exit (Ctrl+O, Enter, Ctrl+X in nano). Then enable and start the service:

# Reload systemd to pick up the new unit file
sudo systemctl daemon-reload

# Enable OpenClaw to start on boot
sudo systemctl enable openclaw

# Start OpenClaw now
sudo systemctl start openclaw

# Verify it's running
sudo systemctl status openclaw

You should see Active: active (running) in green. If it failed, check logs with journalctl -u openclaw -n 50.

Step 6: Configure Nginx Reverse Proxy

Nginx proxies HTTP/HTTPS traffic from port 80/443 to your OpenClaw instance running on port 3000. This enables SSL termination, efficient static file handling, and protection against basic attacks.

# Create nginx config for OpenClaw
sudo nano /etc/nginx/sites-available/openclaw

Paste the following configuration:

server {
    listen 80;
    server_name openclaw.yourdomain.com;

    # Redirect all HTTP to HTTPS (after SSL is configured)
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name openclaw.yourdomain.com;

    # SSL configuration (filled by Certbot)
    ssl_certificate /etc/letsencrypt/live/openclaw.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/openclaw.yourdomain.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Proxy configuration
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 86400;
        proxy_connect_timeout 86400;
        proxy_send_timeout 86400;
    }

    # Block access to sensitive files
    location ~ /\. {
        deny all;
    }

    # Logging
    access_log /var/log/nginx/openclaw_access.log;
    error_log /var/log/nginx/openclaw_error.log;
}

Enable the configuration and test nginx:

# Enable the site (symlink)
sudo ln -s /etc/nginx/sites-available/openclaw /etc/nginx/sites-enabled/

# Remove the default site (avoid conflicts)
sudo rm /etc/nginx/sites-enabled/default

# Test nginx configuration
sudo nginx -t

# Reload nginx
sudo systemctl reload nginx

Step 7: Obtain SSL Certificate

Use Certbot with the nginx plugin to automatically obtain and configure a Let's Encrypt certificate. Certbot handles certificate issuance and renewal.

# Obtain SSL certificate
sudo certbot --nginx -d openclaw.yourdomain.com

# Follow the interactive prompts:
# - Enter email for expiry notifications
# - Agree to Terms of Service
# - Choose whether to share email (optional)
# - Select redirect (type 2) to force HTTPS

Certbot automatically modifies your nginx config to include SSL settings and sets up a cron job or systemd timer for renewal. Let's Encrypt certificates expire after 90 days; Certbot's renewal runs twice daily and only renews when certificates are within 30 days of expiry.

Test the renewal process:

# Simulate renewal (dry run)
sudo certbot renew --dry-run
SSL works without a domain? You can test OpenClaw locally using a self-signed certificate or a tool like localtunnel, but production deployments require a real domain with DNS pointing to your VPS.

Step 8: Verify Everything Works

# Check OpenClaw is running
sudo systemctl status openclaw

# Check nginx is proxying correctly
sudo systemctl status nginx

# Test the SSL endpoint
curl -I https://openclaw.yourdomain.com

# View recent logs
sudo journalctl -u openclaw -n 20 --no-pager

You should see HTTP 200 or a JSON response from OpenClaw at your domain. If you get a 502 Bad Gateway, OpenClaw isn't responding on port 3000 — check that it's running and listening on the correct port.

Step 9: SSH Hardening

Secure SSH access to prevent brute-force attacks:

# Edit SSH config
sudo nano /etc/ssh/sshd_config

Set or verify these settings:

Port 22                         # Change from default if desired
PermitRootLogin no             # Disable root login
PasswordAuthentication no      # Disable password auth (use keys only)
PubkeyAuthentication yes       # Enable key-based auth
MaxAuthTries 3                 # Limit login attempts
ClientAliveInterval 300        # Disconnect idle sessions
ClientAliveCountMax 2

Add your public key to the openclaw user's authorized_keys:

# As openclaw user
mkdir -p ~/.ssh
chmod 700 ~/.ssh
nano ~/.ssh/authorized_keys    # Paste your public key
chmod 600 ~/.ssh/authorized_keys
# Restart SSH
sudo systemctl restart sshd
Test SSH key access before logging out of your current session. Open a new terminal and try ssh openclaw@your-vps-ip. If it works without prompting for a password, your key is configured correctly.

Keeping OpenClaw Running 24/7

Once your systemd service is configured, OpenClaw is already set to survive reboots and recover from crashes. Here's a summary of what keeps it online:

  • systemd Restart policy: Restart=on-failure and RestartSec=10 automatically restart OpenClaw within 10 seconds of any crash
  • Enable on boot: systemctl enable openclaw registers the service in the boot sequence
  • Watchdog: For critical deployments, enable systemd's watchdog to monitor the service and kill/restart if it becomes unresponsive
# Add watchdog support to your service (optional)
# In /etc/systemd/system/openclaw.service, under [Service]:
WatchdogSec=60
Restart=on-failure
# Reload systemd after changes
sudo systemctl daemon-reload
sudo systemctl restart openclaw

Troubleshooting

OpenClaw service won't start

# View detailed logs
sudo journalctl -u openclaw -n 50 --no-pager

# Common causes:
# - Port 3000 already in use: check with `sudo lsof -i :3000`
# - Missing dependencies: run `npm install` in the OpenClaw directory
# - Permission errors: verify openclaw user owns /home/openclaw

502 Bad Gateway in nginx

# Verify OpenClaw is running and responding
curl http://127.0.0.1:3000

# Check nginx error logs
sudo tail -20 /var/log/nginx/openclaw_error.log

# Reload nginx
sudo systemctl reload nginx

SSL certificate not working

# Verify certificate exists
sudo ls -la /etc/letsencrypt/live/openclaw.yourdomain.com/

# Test SSL configuration
sudo certbot certificates

# Force renewal if expired
sudo certbot renew --force-renewal

Domain not resolving

# Check DNS propagation
dig openclaw.yourdomain.com A
nslookup openclaw.yourdomain.com

# Verify A record at your DNS provider
# Propagation can take up to 24-48 hours for new records

Permission denied errors

# Fix OpenClaw directory ownership
sudo chown -R openclaw:openclaw /home/openclaw

# Verify service file paths exist
ls -la /usr/lib/node_modules/openclaw/index.js
ls -la /usr/bin/node

Frequently Asked Questions

Can I run OpenClaw on a Raspberry Pi or home server instead of a VPS?

Yes, technically. OpenClaw runs on any Linux environment with Node.js 22+. However, a home server means your IP is likely dynamic (requiring a dynamic DNS service), your connection lacks the static IPs and reliability guarantees of a VPS, and you'll need to handle port forwarding. For production use, a VPS is strongly recommended. For experimentation, a Raspberry Pi 4 with 4GB+ RAM running Ubuntu Server is a viable low-cost option.

How much does it cost to host OpenClaw per month?

Total monthly cost depends on your VPS choice. At the budget end: a Hetzner CAX11 (2 vCPU, 4GB RAM) at €3.49/month plus domain costs (~$1/month amortized) and LLM API costs (~$5–30/month depending on usage) totals around $10–35/month. At mid-range: Hetzner CAX21 (8GB RAM) at €7.49/month gives you much more headroom. DigitalOcean's comparable plan runs ~$48/month. Add SSL (free via Let's Encrypt), monitoring (free tier UptimeRobot), and you're looking at $10–60/month total depending on provider and API usage.

What happens if my VPS crashes or reboots?

The systemd service configuration in this guide includes Restart=on-failure and systemctl enable openclaw, which means OpenClaw automatically starts on boot and restarts within 10 seconds of any crash. Paired with uptime monitoring (UptimeRobot or Uptime Kuma), you'll get alerted within minutes if anything goes offline. For truly critical deployments, consider a lightweight VPS failover setup or run a secondary instance.

Can I run multiple OpenClaw agents on one VPS?

Yes. Each agent runs as a separate Node.js process. On a CAX21 (8GB RAM), you can comfortably run 2–4 agents depending on their workload and the LLM models they use. Run each as a separate systemd service with a different port. Note that agents sharing resources will compete for RAM and CPU — if agents are doing heavy API calls or local inference, you'll want more headroom.

Do I need a domain name for OpenClaw?

Not strictly — OpenClaw can run on http://localhost:3000 without a domain. However, a domain is required for SSL certificates (HTTPS), which is strongly recommended for any production deployment. A subdomain (e.g., openclaw.yourdomain.com) costs nothing extra if you already own a domain. Some OpenClaw integrations (Telegram bots, OAuth flows) also require HTTPS to function properly.

What's the difference between running OpenClaw via systemd vs. Docker?

systemd is the approach in this guide — it treats OpenClaw as a native system service with proper process management, log handling, and boot ordering. Docker adds an isolation layer and makes it easier to move the service between hosts, but introduces additional complexity (Docker daemon management, networking, volume mounts) for marginal benefit on a single-VPS setup. systemd is simpler and lighter for a one-agent deployment. Docker is better when you need to run multiple services with complex interdependencies or want easy migration.

How do I update OpenClaw to a new version?

If you installed via npm globally (npm install -g openclaw): sudo npm update -g openclaw. Then restart the service: sudo systemctl restart openclaw. Always check the changelog before updating — major version changes sometimes require config file updates. For production agents, test updates on a staging instance first.

Can I use Cloudflare instead of Certbot for SSL?

Yes. If your domain uses Cloudflare as its DNS provider, you can enable Cloudflare's universal SSL certificate (free) which covers your subdomain automatically — no Certbot needed. This also adds Cloudflare's DDoS protection and CDN. The tradeoff is that your traffic is technically passing through Cloudflare's proxy, which may have implications for privacy and data sovereignty depending on your threat model.
